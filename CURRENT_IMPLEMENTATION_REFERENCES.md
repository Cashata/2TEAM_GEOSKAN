# Полезные источники для текущей реализации

Этот файл - курируемая выжимка из локальных Markdown-дампов и учебных примеров. Сырые `repomix-output*.md` остаются в `.gitignore`: они нужны как справочник, но не должны попадать в историю Git, особенно большой `repomix-output.md`.

## Что уже есть в текущем решении

| Файл | Роль |
| --- | --- |
| `fly_orb_ransac.py` | Совместимая CLI-обертка для основного сценария: старые команды запуска остаются рабочими. |
| `geoscan_mission/flight/` | Отдельный слой управления дроном, камерой SDK2/OpenCV, батареей и waypoint-командами. |
| `geoscan_mission/vision/` | ORB/RANSAC-локализация по карте и переиспользуемый ArUco-детектор маркеров задания. |
| `geoscan_mission/trajectory/` | Паттерны `waypoints/square/lawnmower/cube` и grid path planning helpers из учебных файлов. |
| `geoscan_mission/recording.py` | CSV/JSON-строки, видеооверлеи, ArUco-summary и непрерывная запись кадров. |
| `aruco_detector.py` | Compatibility re-export для старых импортов ArUco-детектора. |
| `aruco` | CLI-проверка одного кадра: `python aruco --image frame.jpg --json`. |
| `collect_dataset_mini2.py` | Сбор датасета с камеры для карты, отладки и последующего улучшения локализации. |
| `keypoint_map_localizer.py` | Отдельная локализация кадра на карте по ключевым точкам. |
| `drone_start_assignment_llm.md` | Техническое задание, метрики, ожидаемые артефакты и ограничения миссии. |

Главный вывод: текущая база уже закрывает безопасное распознавание ArUco, запись доказательных логов и привязку маркеров к карте через homography. Следующие полезные шаги - не переписывать это, а аккуратно добавить визуальное наведение, генерацию/проверку маркеров, калибровку камеры и позже захват груза.

## Подробные заметки

### 1. Задание и критерии

`drone_start_assignment_llm.md` - главный источник правды. Из него для текущего кода важны:

- распознавание целевых ArUco ID `3`, `23`, `42`, `117`;
- деление на разрешенные и запрещенные цели;
- восстановление слова по найденным буквам;
- доказательные логи и возможность объяснить, где и когда найден каждый маркер;
- аккуратная работа с зоной 3 x 3 м и запретными объектами.

Текущий `fly_orb_ransac.py --aruco` уже пишет `aruco_seen_ids`, `aruco_new_ids`, `aruco_word`, `aruco_allowed_ids`, `aruco_forbidden_ids`, `aruco_markers_json`. Это хорошая база для отчета и демо, потому что результаты можно проверить по CSV/JSON без просмотра всего видео.

### 2. ArUco-детекция

Самые полезные учебные примеры - `11_aruco_basics_example1_detect.py`, `auto_detect_marker.py`, `detect_aruco.py` и `detect_aruco_coordinates.py`.

Что уже применено:

- детектор вынесен в отдельный `ArucoDetector`, а не размазан по полетному сценарию;
- есть fallback для разных OpenCV API;
- найденные ID накапливаются между кадрами;
- бизнес-логика задания находится в словаре целей, а не в условных блоках внутри цикла полета.

Что стоит добавить следующим:

- debug-команду для перебора словарей, чтобы быстро понять, почему конкретная распечатка не детектится;
- helper для генерации маркеров нужных ID и словаря;
- опциональную pose-оценку через `solvePnP`, когда появятся реальные `camera_matrix` и `dist_coeffs`.

### 3. Полет и поиск маркеров

`ar.py`, `aruco_flight.py` и `aruco_grid_patrol.py` полезны как набор идей, но не как готовая замена текущему `fly_orb_ransac.py`.

Полезные идеи:

- держать камеру и управление полетом раздельно, чтобы обработка кадра не тормозила команды;
- считать несколько стабильных кадров перед принятием решения;
- использовать yaw-поиск, если маркер временно потерян;
- иметь простой grid patrol как fallback-маршрут;
- логировать FPS и задержки камеры.

Для текущей реализации лучше добавить это как новый слой поверх имеющейся ORB/RANSAC-локализации: сначала летим и пишем карту/логи, затем при обнаружении нужного маркера можно включать короткую фазу выравнивания.

### 4. Координаты маркеров

Сейчас координаты ArUco в `fly_orb_ransac.py` вычисляются так:

- pixel center/corners сохраняются всегда;
- если homography валидна, центр маркера проецируется в координаты карты 3 x 3 м;
- если homography невалидна, координаты карты остаются `null`.

Это безопаснее, чем притворяться, что у нас есть точная 3D-поза без калибровки. Примеры `11_aruco_basics_example2_coordinates.py` и `aruco_flight.py` пригодятся для следующего шага: добавить `solvePnP` только после того, как будет файл калибровки камеры и известный физический размер маркера.

### 5. Груз, захват и доставка

`cargo_mission.py`, `ikar_bas_mission.py` и Lua-примеры с cargo/magnet полезны для будущей итерации. Из них стоит взять:

- единый `cargo_act(grab: bool)`;
- fallback через RC-канал, если SDK-метод недоступен;
- выравнивание по центру цели перед захватом/сбросом;
- ToF-проверку высоты перед снижением;
- precheck батареи перед стартом.

В текущую реализацию это пока не стоит смешивать с ArUco-распознаванием. Сначала нужно стабильно находить цели и писать координаты, потом добавлять actuator-слой.

### 6. Камера, датасеты и калибровка

`camera_calibration.py`, `19_map_creation_example1_calibration.py`, `19_map_creation_example2_generation.py`, `19_map_creation_example3_builder.py` и `19_map_creation_example6_vector_generation.py` закрывают три полезные задачи:

- калибровка камеры для `solvePnP`;
- печать маркеров нужного словаря;
- сбор/сборка карты для более надежной локализации.

В текущем репозитории это соответствует `collect_dataset_mini2.py`, `keypoint_map_localizer.py` и параметру `--reference` в `fly_orb_ransac.py`. Самый практичный следующий артефакт - короткий script/helper для генерации маркеров миссии и README-инструкция, какой словарь используется.

### 7. Телеметрия и безопасность

`GET_battery_status.py`, `GET_dist_sensor_data.py`, `GET_local_position.py` и разделы `13_camera_advanced.tex`, `14_telemetry.tex` полезны как справочник по проверкам перед полетом и дополнительным логам.

Ближайшие улучшения:

- явно логировать напряжение батареи в начало JSON/CSV или отдельный session summary;
- добавить ToF-высоту перед потенциальным снижением к цели;
- при доступном LPS писать локальную позицию рядом с map-позицией, чтобы сравнивать две системы координат.

### 8. Multi-drone и swarm

Group-flight примеры пока не надо тащить в код. Они полезны как материал для будущего координатора трех дронов: распределение маршрутов, роли, избегание дублирования целей. Для текущего single-drone прогона достаточно, чтобы каждый запуск писал полные логи, а объединение результатов можно сделать отдельным постпроцессингом.

### 9. Что не использовать сейчас

- Большой `repomix-output.md` читать только точечным поиском: там много повторов, веб-симулятора, старых уроков и нецелевого материала.
- HumanTracking, PyCharm stubs, сгенерированные assets и UI-части симулятора не помогают текущему Python flight stack.
- Lua-примеры полезны для понимания onboard API и cargo/magnet, но текущий код написан на Python SDK2, поэтому прямое копирование только увеличит риск.
- Примеры на старом `pioneer_sdk` надо адаптировать к текущему стилю `pioneer_sdk2`, `create_pioneer`, `Sdk2Camera`, `CameraType.OPT`, timeout-оберткам и явному `camera.close()`.

## Рекомендуемая очередь внедрения

1. `P0`: добавить маленький helper/CLI для генерации маркеров `3/23/42/117` тем же словарем, что в `aruco_detector.py`.
2. `P0`: добавить debug-режим проверки словаря ArUco на одном изображении, чтобы быстро диагностировать чужие распечатки.
3. `P1`: добавить опциональный calibration-файл и pose-оценку маркера через `solvePnP`; оставить текущую homography-проекцию как fallback.
4. `P1`: вынести визуальное выравнивание по центру маркера в отдельный модуль/фазу, включаемую явным флагом.
5. `P2`: после стабильной детекции добавить cargo actuator-слой с RC fallback и ToF-защитой.
6. `P2`: сделать postprocess-скрипт для объединения логов нескольких дронов, если потребуется командная миссия.

## Практическое правило

Для текущего состояния проекта полезны не все учебные примеры, а только те, которые усиливают уже выбранную архитектуру: надежное распознавание, понятные координаты, проверяемые логи, безопасная телеметрия и минимальные опциональные режимы. Все, что требует новой полетной архитектуры или actuator-команд, лучше добавлять отдельными флагами после полевых smoke-тестов.
